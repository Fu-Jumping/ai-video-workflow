# Capability Index

## 目的

本文件列出六步主流程之外，推荐随工作流一起建设的横切能力。

这些能力不是新的步骤，而是帮助多轮修改、多人协作和系统化交付保持稳定的配套层。

它同时要求：若目标 IDE 的运行入口依赖目录化 skill，则除了流程能力本身，还必须把可识别 skill 包同步到原生入口，而不是只保留文档镜像。

本包所说的“增强流程”，就是在六步主流程之外，默认随工作流一起启用的以下能力：

- 审核与通过机制
- 影响分析与变更传播
- 修复与回链流程
- 交付总控

默认情况下，AI 初始化时不应只读六步主流程，还必须额外读取本文件。

只有项目显式关闭增强流程时，AI 才能不按本文件执行；否则应默认按已启用处理，避免因用户未明确提到而漏掉安全相关检查。

## 1. 审核与通过机制

- 用途：把“生成完成”和“可作为正式依赖”区分开
- 典型动作：
  - 通过
  - 打回修改
  - 拒绝
  - 记录审核备注

## 2. 影响分析与变更传播

- 用途：在上游修改后判断哪些下游内容失效
- 典型动作：
  - 记录依赖快照
  - 标记受影响项
  - 输出复查或重做建议
- 推荐触发：
  - 已完成步骤中的局部细节被修改
  - 已完成步骤中的一个或多个文件被修改
  - 已通过审核的上游事实被改写
- 默认回报方式：
  - 先以对话回报检查结果
  - 必要时再补写日志或总控文件

## 3. 修复与回链流程

- 用途：在冲突出现后给出修复顺序，而不是让下游偷偷硬补
- 典型动作：
  - 暴露冲突点
  - 推荐回修顺序
  - 区分复查与重做
- 默认要求：
  - 不静默修改下游
  - 不把回链修复伪装成普通局部编辑
  - 优先把回修顺序明确告诉用户

## 4. 交付总控

- 用途：在交付前统一查看缺失项、阻塞项和素材完备度
- 典型动作：
- 汇总缺失项
- 汇总阻塞项
- 汇总图片 / 视频 / 交付状态
- 按需补出锚点统筹、锁版清单和重复主体治理文档
- 若进入多人执行，补出执行人分配、连续范围切分、样张先行与批量放行状态

## 5. 统一对话回报约定

增强流程中的关键检查默认采用“对话回报优先”：

- 影响分析结果先回到对话，再决定是否扩散修订
- 审核打回或拒绝先回到对话，再进入修订
- 总控发现缺失项或阻塞项先回到对话，再进入补齐

对话回报至少应包含：

- 触发原因
- 修改或检查范围
- 上游核对结论
- 当前步骤联动结论
- 下游影响结论
- 建议处理顺序
- 是否存在阻塞或需要用户确认

并且在每次模型回复末尾，都应继续附带统一状态回报：

- 当前状态
- 当前步骤
- 触发源步骤（仅增强流程时）
- 本次回复完成了什么
- 接下来可以做什么

默认使用紧凑版状态回报；只有在增强流程、正式完成态、审核结论或阻塞场景下，改用完整版。

如果没有发现扩散影响，也应明确回报“未发现需要扩散处理的上游或下游问题”。

## 6. 推荐使用方式

- 最小版工作流：只启用六步主流程；仅在项目显式关闭增强流程时使用
- 增强版工作流：六步主流程 + 审核 + 影响分析 + 修复 + 总控；本包默认模式
如果团队只做单人、单轮、低复杂度项目，也应通过显式配置后再退回最小版；如果项目会频繁修改，应保持默认增强版。

无论使用哪一种模式，只要 IDE 依赖目录化 skill 识别，就必须同时维护：

- 长文规则源
- 目录化 skill bundle
- 原生运行入口中的 skill 包
