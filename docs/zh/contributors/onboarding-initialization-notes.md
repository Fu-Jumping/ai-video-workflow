# 新手启动与初始化体验记录

本文记录当前关于 `ai-video-workflow` 新手启动、初始化入口和 CLI 防呆机制的产品判断。它不是实施计划，不包含任务切片、提交切片或验收清单。

## 问题背景

当前项目已经有 CLI、官方 pack、校验、诊断、IDE runtime 同步、Obsidian 观看层和 MCP 只读上下文。但对完全没有编程经验的创作者来说，最早的使用路径仍然不够直白。

典型困惑包括：

- 用户不知道应该通过 `git clone`、Download ZIP，还是其他方式把工具仓库放到本地。
- 用户可能把 `ai-video-workflow` 工具仓库误当成自己的创作项目。
- 用户不知道下载后第一步应该做什么。
- 用户不知道真正应该打开哪个目录给智能体工作。
- 用户不知道第一句话应该如何对智能体说。
- 用户看到 CLI 参数、路径、空格、平台枚举值时容易输错。
- 用户不清楚 Obsidian 观看层、智能体工作目录和 Step 1 到 Step 6 源文件之间的关系。

因此，当前缺口不是单个命令缺失，而是“工具仓库入口”和“创作项目入口”没有被足够清晰地拆开。

## 当前入口状态

当前推荐模型实际上是：

1. 先下载或克隆 `ai-video-workflow` 工具仓库。
2. 在工具仓库内安装依赖并构建 CLI。
3. 使用 CLI 生成真正的创作项目。
4. 让 Codex、Cursor、Claude Code、Trae 或其他智能体打开生成后的创作项目目录。
5. 从 Step 1 到 Step 6 推进创作。

CLI 初始化会生成项目骨架，并同步共享智能体入口和指定 IDE 的 runtime 目录。它不仅是创建文件夹，还负责把 `AGENTS.md`、`docs/ai-workspace/`、pack skills、templates、indexes 和 IDE 规则放到正确位置。

所以从当前实现看，CLI 是标准初始化入口。手工复制理论上可行，但对普通创作者不现实，也容易漏掉运行规则、模板或校验边界。

## 人类入口与 CLI 入口

后续产品表达应分成两层：

- 人类新手入口：告诉用户“你只需要下载工具仓库，然后让智能体帮你创建创作项目”。
- CLI 标准入口：保留确定性的项目生成、同步和校验能力，供智能体、熟练用户、测试和 CI 使用。

新手文档不应该默认要求用户理解 `pnpm install`、`pnpm build`、`node apps/cli/dist/index.js init` 这些命令。更合理的表达是：智能体可以代替用户检查工具仓库、询问项目参数、运行初始化命令，并解释下一步。

## 可复制的第一句话

新手入口应提供一段可以直接复制给智能体的话：

```text
我想用当前 ai-video-workflow 工具创建一个新的 AI 视频创作项目。
请先检查这个工具仓库是否已经安装和构建；如果没有，请帮我完成。
然后询问我项目名称、项目保存位置、使用哪个智能体工具、默认图片生成平台和默认视频生成平台。
请你根据我的回答运行初始化命令，创建项目后告诉我应该打开哪个目录，并带我从 Step 1 策划开始。
我没有编程经验，请不要让我自己拼复杂命令；每一步只告诉我需要确认或填写什么。
```

这段话的目标不是替代 CLI，而是让智能体成为 CLI 的低门槛人机界面。

## CLI 参数防呆判断

当前 CLI 已经有一部分防呆能力：`init` 在缺少参数时会进入交互式选择，用户不用手写 IDE、图片平台和视频平台枚举值。

但仍然存在明显风险：

- 用户手动输入 `--ide codx`、`--image open-ai` 这类拼写错误时，当前只会精确匹配失败。
- 用户复制路径时，如果路径包含空格，可能忘记加引号。
- 用户可能在工具仓库根目录里直接生成创作项目，之后误把工具仓库和创作项目混在一起。
- 用户可能不知道 `--name` 是目录名，不知道是否允许中文、空格或特殊字符。
- 用户可能不知道非交互参数适合脚本和智能体，不适合完全新手手敲。

这些问题有必要加防呆，但不必把所有复杂性都暴露给人类用户。

## 建议的防呆方向

优先级较高的方向：

- 让无参数 `init` 成为明确的向导入口，文档中优先展示 `ai-video-workflow init`，少展示完整长命令。
- 当用户传入非法 `--ide`、`--image` 或 `--video` 时，输出“可选值”和“你是不是想用某个值”的提示。
- 对项目名称做目录名检查，提示中文、空格、特殊字符、已有目录和空目录风险。
- 如果检测到用户正在工具仓库内创建创作项目，提示“工具仓库不是创作项目”，并建议选择单独的创作项目父目录。
- 增加初始化完成后的下一步输出：项目路径、应该打开哪个目录、应该对智能体说什么、如何运行 `verify`。
- 对包含空格的路径，在输出命令示例时自动加引号。
- 在执行写入前给出摘要确认：项目名、保存位置、IDE、图片平台、视频平台、将写入的主要目录。

可以后置的方向：

- 支持平台和 IDE 的别名，例如 `claude` 映射到 `claude-code`。
- 支持轻量拼写纠错，例如 `codx` 提示 `codex`。
- 增加 `init --guided` 作为显式新手向导别名。
- 增加 `init --parent <path>` 或类似参数，把“项目父目录”和“项目名称”拆开，减少路径拼接错误。

## 目录与顺序组合风险

初始化的高风险点不只是参数输错，还包括“智能体当前在哪个目录”“用户想把创作项目放在哪里”“目标目录是否已经有东西”“其他智能体是否已经写入宿主文件”等排列组合。

下面按当前实现记录风险。这里的“工具仓库”指 `ai-video-workflow` 本身，“创作项目”指 CLI 生成后包含 Step 1 到 Step 6 的项目目录。

这些风险可以按链路粗分为十段：

- 获取工具仓库：用户可能把工具仓库误认为创作项目。
- 选择工作目录：智能体可能停留在工具仓库、创作项目父目录、空创作目录或已有产物目录。
- 初始化：`init` 当前以 `process.cwd()` 作为父目录，并把 `--name` 直接拼成目标路径。
- 生成后入口：新项目生成后仍缺少真正面向创作者的第一步文件和下一步提示。
- 重复操作：已经初始化过的目录再次 `init` 时，用户不容易看出哪些文件会被覆盖。
- 同步 runtime：`sync` 当前也会在不存在的路径上创建共享入口和 runtime 表面。
- 校验诊断：`verify/doctor` 对路径不存在和非法 IDE 的错误不够友好。
- Obsidian 观看层：导出和校验会先追求投影自洽，源项目不存在时可能生成空观看层。
- MCP/new-pack：MCP 当前偏只读安全；`new-pack --name` 和 `init --name` 有同类目录名风险。
- 文档导航：README、quickstart、VitePress sidebar 和版本叙事还没有完全收敛到同一条新手路径。

| 场景 | 当前风险 | 建议判断 |
| --- | --- | --- |
| 智能体当前在工具仓库，用户想创建新项目 | 如果直接运行 `init --name my-film`，项目会生成在工具仓库下面，用户可能把工具仓库和创作项目混在一起 | 允许但不推荐；向导应提示选择单独的创作项目父目录 |
| 智能体当前在创作项目父目录，目标子目录不存在 | 这是最清晰的路径 | 推荐默认路径 |
| 智能体当前已经在一个空创作目录里 | 当前 `init --name xxx` 会再创建一层 `xxx/`，可能多嵌套一层 | 需要支持或说明“在父目录运行 init”，或后续增加“初始化当前目录”模式 |
| 用户传入中文或含空格项目名，并正确加引号 | 烟测确认 `--name "中文 项目"` 可正常创建 | 可以支持；但输出示例必须自动加引号，避免用户复制路径时拆词 |
| 用户传入空项目名 `--name ""` | 当前会把 `process.cwd()` 本身初始化为项目，并输出空的 `Created` | 高风险；空名称应禁止，当前目录模式必须显式开关 |
| 用户传入全空格项目名 | 当前会创建难以识别的空格目录，并输出不清晰的 `Created` | 高风险；应 trim 后校验，禁止空白首尾和不可见名称 |
| 用户传入尾随点、尾随空格或 Windows 保留名，例如 `trailing-dot.`、`trailing-space `、`CON` | 烟测确认当前都会被接受，其中保留名会给后续清理和跨平台访问制造麻烦 | 高风险；项目名应校验 Windows/macOS/Linux 交集安全名称 |
| 用户传入 `--name .` | 当前会直接把当前工作目录初始化为项目，且输出 `Created .`；如果当前目录是工具仓库、素材目录或智能体已有工作目录，会把项目文件写进去 | 高风险；应明确禁止或改成显式 `--current-dir` 模式并要求确认 |
| 用户传入 `--name ..\outside` 或带路径分隔符的名称 | 当前会把项目写到目标根目录外，实测 `..\escape` 会逃到父目录 | 高风险；`--name` 应只允许安全目录名，不能包含路径分隔符、`.`、`..` 或绝对路径片段 |
| 用户传入 `--name nested\demo` | 当前会创建嵌套目录，不一定符合“项目名称”的语义 | 中风险；如要支持嵌套，应拆分为 `--parent` 和 `--name` 并在摘要中显示最终路径 |
| 用户把绝对路径误填进 `--name`，例如 `C:\escape` | 当前会拼成目标父目录下的非法路径并输出 EINVAL stack | 应提示用户把父目录和项目名分开填，不允许 `--name` 包含盘符、UNC 或绝对路径 |
| `init` 成功后用户进入项目 | 当前 Step 1 到 Step 5 目录为空，Step 6 三个文件为空，项目 README 只有 `Generated by ai-video-workflow.` | 高风险；生成项目应包含创作者首页、Step 1 起步文件或清晰指向 runtime templates 的下一步 |
| 目标同名目录已存在且里面有用户文件 | 当前 `init` 会进入该目录继续写入，部分文件可能被覆盖 | 高风险；应先检查非空目录并要求确认或中止 |
| 目标同名目录已经是 Git 仓库 | 烟测确认存在 `.git` 不会阻止 `init` 写入配置、README 和 runtime | 高风险；应把 `.git` 目录视为“非空且重要”的强提示，默认拒绝或要求显式确认 |
| 目标目录已有 `AGENTS.md` | `sync` 使用缺失才写策略，不会覆盖已有 `AGENTS.md` | 相对安全；但需要提示用户按 `ENTRYPOINT_RECONCILIATION.md` 合并 ai-video-workflow block |
| 目标目录已有 `docs/ai-workspace/*` | 已存在文件不会被覆盖，缺失文件会补齐 | 相对安全；但旧版本内容可能不会自动升级 |
| 目标目录已有 `.codex/`、`.cursor/`、`.claude/`、`.trae/` | runtime mirror 复制使用覆盖写入，可能覆盖这些目录下同路径的生成内容 | 中风险；这些目录应视为可重建运行表面，但如果用户手写过同名文件，应先提示 |
| 目标目录已有 Cherry Studio `SOUL.md`、`USER.md`、`memory/` | 当前共享规则把它们视为宿主/用户表面，并加入忽略；`sync` 不生成或覆盖它们 | 当前方向正确；仍需提示智能体不要把这些内容当成 Step 源文件 |
| 目标目录已有自定义 `AGENTS.md`、`SOUL.md`、`USER.md`、`memory/` 后再运行 `init` | 烟测确认宿主文件会保留，但 `verify` 会提示自定义 `AGENTS.md` 需要合并 ai-video-workflow shared entry block | 保留宿主文件是正确行为；向导应在初始化前说明后续可能需要合并入口规则 |
| 目标目录已有根 `.obsidian/` 或 `_views/` | 校验已倾向忽略这类本地/观看层表面，但初始化本身不应把它们当源文件 | 应提示用户项目根不是 Obsidian vault，Obsidian 只打开 `_views/obsidian/` |
| 已经初始化过，又重复运行 `init` | 可能重写 `project.config.yaml`、项目 README，并覆盖 runtime mirror 中的 pack 文件 | 高风险；重复操作应优先使用 `sync`，不是再次 `init` |
| 已初始化为 Codex 项目后，再 `sync --ide cursor` | 烟测确认 Codex 和 Cursor 两套 runtime 都可验证通过，`project.config.yaml` 仍保留原 `ide: codex` | 多 IDE 同步本身可行；但配置中的单一 `ide` 字段会让用户误以为只有一个 IDE |
| 已初始化项目后，再用另一个 IDE 重复 `init` | 烟测确认会重写 `project.config.yaml`，留下旧 runtime，并新增新 runtime；两个 IDE verify 都可通过 | 高风险；重复 init 会改变配置事实并留下多套 runtime，应该改用 sync 或显式迁移 |
| 在一个已初始化项目内部再运行 `init --name child-project` | 烟测确认会创建嵌套子项目，父项目 `verify` 仍通过 | 高风险；应拒绝在项目根内部创建另一个项目，或至少强提示嵌套项目不属于父项目 Step 源 |
| 已经开始写 Step 文件后重复运行 `init` | Step 目录会保留，部分执行计划空文件通常不会覆盖已有内容，但配置、README、runtime mirror 仍有覆盖风险 | 应要求先运行 `verify/doctor`，只缺 runtime 时运行 `sync` |
| 用户把剧本素材先放进目录，再初始化 | 如果素材不与生成文件同名，通常会保留；如果与 README、配置、runtime 路径同名，有覆盖风险 | 向导应先盘点目录，再建议迁移素材或选择新目录 |
| 用户运行 `sync --project <不存在路径>` | 当前会创建一个只有 `AGENTS.md`、`docs/ai-workspace/`、`.gitignore` 和 IDE runtime 的半成品目录，没有 `project.config.yaml` 和 Step 目录 | 高风险；`sync` 应拒绝不存在项目，或要求显式 `--create-missing-runtime-root` 之类的高级开关 |
| 用户运行 `sync --ide` 拼错，例如 `codx` | 当前命令会显示 `Sync complete`，但只写共享入口和 `.gitignore`，不会写 IDE runtime | 高风险；所有接收 `--ide` 的命令都应统一校验枚举值 |
| 用户运行 `init --ide/--image/--video` 拼错 | 当前能阻止写入，但错误以 Node stack trace 形式输出，不是面向新手的 CLI 提示 | 应统一捕获 action error，只输出错误摘要、可选值和下一步建议 |
| 用户运行 `verify/doctor --ide` 拼错 | 当前会触发运行时 TypeError，而不是友好错误 | 高风险；应在 CLI 边界校验并输出可选值 |
| 用户运行 `verify/doctor --project <不存在路径>` | 当前会报一串缺失文件，而不是直接说项目目录不存在；`doctor` 还会建议 rerun `init` 或 `sync` | 中高风险；应先检查项目根是否存在且是目录，再做结构诊断 |
| 用户把 `--project` 指到一个普通文件 | `verify` 会 ENOTDIR stack，`sync` 会 EEXIST stack，`mcp-context` 会报缺配置 stack，`mcp-server` 源码上复用同一 context builder，`export-obsidian` 还会生成空观看层 | 高风险；所有接收 project root 的命令都应先断言“存在且是目录” |
| `project.config.yaml` 内容损坏或枚举值错误 | YAML 语法错误会变成 stack trace；`pack`、`ide`、默认平台填成未知值时，`verify` 仍可通过 | 中高风险；应引入配置 schema 校验并把非法值纳入 doctor |
| 项目只有部分 Step 目录或少量 Step 文件 | 烟测确认 `verify` 会失败，但 `export-obsidian` 仍可基于现有文件生成投影 | 高风险；导出前应要求项目级 verify 通过，或显式允许 partial export 并在输出中标红 |
| 用户运行 `export-obsidian --project <不存在路径>` | 当前会成功生成一个空 Obsidian 观看层 | 高风险；导出前应验证项目根和 Step 目录，不能对空项目生成“看似成功”的 vault |
| 用户随后运行 `verify-obsidian --project <不存在路径> --vault <空观看层>` | 当前可能通过校验，因为 manifest 和生成文件内部自洽 | 高风险；`verify-obsidian` 应验证源项目存在、Step 目录存在，且 manifest 至少对应真实源文件或明确允许空项目 |
| 用户省略 `export-obsidian --out` 或 `verify-obsidian --vault` | 当前会输出 uncaught stack trace；测试只验证包含错误文本，没有验证友好格式 | 应在 commander 层使用可读错误，不打印 Node stack |
| 用户把 Obsidian `--out/--vault` 指到普通文件 | `export-obsidian` 会 EEXIST stack，`verify-obsidian` 会 ENOTDIR stack | 应先校验目标类型；普通文件应直接报“目标不是目录” |
| 用户把 Obsidian `--out` 指到项目内部普通子目录 | 当前主要禁止项目根和项目父目录，项目内部非 `_views/obsidian` 子目录仍需单独确认 | 高风险；观看层不应混入 Step 源文件树，除明确的 `--in-project-view` 外应拒绝项目内部输出 |
| 用户对项目内部普通子目录使用 `export-obsidian --force --out` | 如果目标没有 `.git`，可能删除项目内用户资料或素材 | 高风险；`--force` 应识别项目内输出风险，只有 `_views/obsidian` 可作为内置观看层目标 |
| 用户对含 `.git` 的 vault 执行 `--force --dry-run` | 当前 dry-run 会正常列出创建计划，不提示真实 force 会被 `.git` 阻止 | dry-run 也应执行安全预检并标记危险原因 |
| 用户阅读中英文 README/quickstart/VitePress 导航 | 中文 README 已有新手入口，但英文 README 没有；中英文 quickstart 仍从 `pnpm install/build` 开始；新记录文档只在中文首页链接，未进入 sidebar | 应统一中英文新手入口、侧边栏入口和“让智能体代跑 CLI”的默认叙事 |
| 用户看到版本叙事 | package 仍是 `0.1.0`，README 写 v0.2，Obsidian 文档写 v0.3 到 v0.7，容易给人“版本跳太快”的感觉 | 应把路线图版本、功能批次和实际包版本关系讲清楚，后续体验迭代不应继续抬高版本叙事 |
| 用户运行 `sync --project <工具仓库根>` 或 `sync --project apps/cli` | `sync` 会把共享入口和 runtime 写入传入目录；这与“工具仓库不是创作项目”的边界冲突 | 高风险；`sync` 也应识别工具仓库/源码目录，提示先创建或选择真实创作项目 |
| 用户运行 `init --name <已存在普通文件名>` | 目标路径不是目录时会落到底层文件系统错误 | 应在写入前给产品级错误，说明目标已存在但不是目录，并保证不创建旁路文件 |
| 新项目结构校验通过，但创作者仍没有可填写的 Step 1 内容 | 结构正确不等于新手能开始创作；当前生成 README 和 Step 文件过空 | 应把“结构验证”和“创作入口可用性”分开，生成项目必须提供第一步内容或模板指引 |

## 组合方式当前结论总览

这里把上面的场景压成“当前实现会不会出错”的决策表，便于后续封堵时逐项对照。

| 组合方式 | 当前是否会出错 | 结论 |
| --- | --- | --- |
| 工具仓库中检查 CLI，然后切到创作项目父目录运行 `init --name <safe-name>` | 不会出错 | 推荐默认路径 |
| 在创作项目父目录生成不存在的目标子目录 | 不会出错 | 推荐默认路径 |
| 生成后切到项目根，让智能体读取 `AGENTS.md` 和 `docs/ai-workspace/` | 不会出错 | 推荐默认路径 |
| 同一项目补同步第二个 IDE runtime | 不会出错，但配置表达不完整 | 可支持，但需要解释多 IDE 与 `config.ide` 的关系 |
| 已有自定义 `AGENTS.md` 或 Cherry Studio 宿主文件后初始化/同步 | 不会覆盖宿主文件，但需要合并入口规则 | 可支持，但必须提示人工合并 |
| 在工具仓库、空创作目录、已有项目目录中直接运行 `init` | 命令可能成功，但目录层级或写入位置可能错 | 当前高风险 |
| 项目名为空、全空白、`.`、`..`、带分隔符、绝对路径、保留名、尾随空格/点 | 可能成功写错位置，或以 stack 失败 | 当前高风险，应统一禁止 |
| 目标目录非空、已有 Git 仓库、已有 README/config/runtime | 命令可能成功，但会覆盖或混入生成文件 | 当前高风险 |
| 重复 `init`，尤其换 IDE 重复 init | 命令成功，但重写配置并保留旧 runtime | 当前高风险 |
| 在项目内部创建另一个项目 | 命令成功，父项目校验仍可能通过 | 当前高风险 |
| 先 `sync` 不存在目录，再 `init` | `sync` 先生成半成品 | 当前高风险 |
| `verify/doctor/sync` 使用非法 IDE | 可能假成功或 TypeError | 当前高风险 |
| `--project` 指向不存在路径或普通文件 | 诊断不清，部分命令 stack，导出还可能成功 | 当前高风险 |
| 配置 schema 或枚举值损坏 | 可能 stack 或 verify 误通过 | 当前高风险 |
| 部分项目、空项目、缺 Step 目录项目导出 Obsidian | 可能生成看似成功的投影 | 当前高风险 |
| Obsidian 输出目标为项目根、项目父目录或含 `.git` 的 force 目标 | 部分已有保护 | 当前相对安全，但 dry-run 和普通文件目标仍需补齐 |
| MCP context/server 指向有效项目 | 不会写文件 | 当前安全 |
| `new-pack --name` 使用不安全名称 | 与 `init --name` 同类风险 | 当前高风险 |

## 当前覆盖语义

按当前实现，初始化和同步大致分三类写入：

- 缺失才写：`AGENTS.md`、`docs/ai-workspace/*`、`CLAUDE.md` 这类共享或平台入口会尽量保留已有文件。
- 追加或补块：`.gitignore` 会追加 ai-video-workflow 的本地/生成表面忽略块。
- 覆盖写入：pack skills、templates、workflow indexes、IDE runtime mirror、`project.config.yaml` 和当前生成项目 README 存在覆盖风险。

本轮烟测确认：如果目标同名目录已有用户写的 `README.md` 和 `project.config.yaml`，再次运行 `init --name <同名目录>` 后，这两个文件会被覆盖成 ai-video-workflow 生成内容。

这个覆盖语义对“由 CLI 管理的运行镜像”是合理的，但对新手来说不可见。因此后续向导应在写入前显示摘要：

```text
将创建或更新：
- 创作项目根目录
- Step 1 到 Step 6 目录
- AGENTS.md 和 docs/ai-workspace/
- 目标 IDE 的 runtime mirror

可能覆盖：
- project.config.yaml
- README.md
- .codex/.cursor/.claude/.trae 下的 ai-video-workflow 生成文件
```

如果检测到目标目录不是空目录，应先询问用户是“继续补齐现有项目”“只同步智能体规则”“另建新项目”，不要直接默认初始化。

## 已有保护机制

本轮排查确认当前已经具备这些保护，后续不应回退：

- `init` 的 `--ide`、`--image`、`--video` 在显式传参时会做枚举校验；缺参时会进入交互选择。
- `sync` 不覆盖已有 `AGENTS.md`、`docs/ai-workspace/*` 和 `CLAUDE.md`。
- `sync` 会把 `_views/`、`.obsidian/`、`.codex/`、`.cursor/`、`.claude/`、`.trae/`、`SOUL.md`、`USER.md`、大小写变体和 `memory/` 写入 `.gitignore` 保护块。
- `verify` 会忽略根 `_views/`、根 `.obsidian/`、Cherry Studio `SOUL.md`、`USER.md` 和 `memory/`，避免本地观看层或宿主记忆污染项目级 Markdown 链接检查。
- `verify` 能识别自定义 `AGENTS.md` 缺少 ai-video-workflow block 的情况，并要求合并而不是覆盖。
- `doctor` 对自定义 `AGENTS.md` 会输出可合并 block，并提醒不要复制 Cherry Studio 私有记忆、token、本机路径或平台缓存到项目事实源。
- Obsidian 默认增量导出会保留 `Notes/` 用户笔记，并跳过用户手改过的生成文件。
- Obsidian `--force` 会拒绝删除包含 `.git` 的输出目录。
- Obsidian 导出禁止输出目录等于项目根或项目父目录。
- Obsidian manifest 和 MCP context 已有“不写本机绝对路径”的测试约束。
- MCP tools 当前只暴露只读工具，不暴露写文件、sync、export-obsidian 或 LibTV 执行工具。

### 额外高风险组合

除上表外，本轮还应把下面组合纳入后续封堵视野：

- 用户把 `ai-video-workflow` 工具仓库克隆进一个已经用于创作的工作区，然后在工具仓库内直接 `init`，导致工具仓库和创作项目嵌套。
- 用户先让智能体写了一批草稿、素材索引或 Cherry Studio 宿主文件，再运行 `init`，其中同名 `README.md`、`project.config.yaml` 或 runtime 路径存在覆盖风险。
- 用户已经有一个半成品 ai-video-workflow 项目，但缺 runtime，于是重复 `init` 而不是 `sync`，导致配置被重写。
- 用户把 Obsidian 外部 vault、项目根、项目父目录、`_views/obsidian/` 或其他已有知识库目录互相混用，导致观看层、源文件和个人笔记边界不清。
- 用户把 `export-obsidian --out` 指向项目内部普通子目录，尤其再叠加 `--force`，可能把观看层混进源文件树或删除项目内用户资料。
- 用户把 `sync --project` 指向工具仓库根、`apps/cli` 或其他源码目录，导致 runtime 和共享入口写入工具代码区。
- 用户把 `--project`、`--out`、`--vault` 指向普通文件、快捷方式、联接目录、只读目录或网络路径，当前大概率表现为底层异常而不是产品级诊断。
- 用户把 `init --name` 填成已有普通文件名，当前应被视为目标类型错误，而不是让底层 `ensureDir` 或写入流程报错。
- 用户复制含空格路径但没有加引号，命令行会把路径拆成多个参数；文档示例和智能体输出需要默认给路径加引号。
- 用户在多个智能体中同时打开同一个目录，一个智能体运行 `sync/export-obsidian`，另一个智能体手改 runtime 或观看层，可能产生难以解释的漂移。
- 用户把 `new-pack` 当成创作项目初始化入口，或把 pack scaffold 与创作项目混在一起；`new-pack --name` 已确认存在与 `init --name` 同类路径风险。
- 新创建项目即使结构校验通过，也可能因为 README 和 Step 1 入口过空而无法让创作者实际开始；后续应把“结构正确”和“新手可启动”作为两个不同验收面。

## 测试覆盖缺口

现有测试覆盖了许多 adapter 边界，但还缺少下面这些回归用例：

- `init` 拒绝 `--name .`、`--name ..`、带路径分隔符和空白首尾名称。
- `init` 拒绝空项目名、全空格项目名和不可见目录名。
- `init` 拒绝 Windows 保留名、尾随点、尾随空格、盘符、UNC 路径和绝对路径片段。
- `init` 对非空目标目录默认失败，并在显式确认或 force 模式下才写入。
- `init` 对已有 `.git` 的目标目录默认失败，或要求显式确认。
- `init --name <已存在普通文件名>` 输出明确错误，并保证不写入任何项目文件。
- `init` 在已有 ai-video-workflow 项目内部运行时默认失败，避免嵌套项目。
- `init` 不覆盖已有 `README.md`、`project.config.yaml` 或用户 Step 文件，除非显式授权。
- `init` 生成项目后包含可用的新手首页、Step 1 起步文件或模板引用，并校验这些文件非空。
- `sync --ide <second-ide>` 可以作为多 IDE runtime 同步通过，但不改变主配置；重复 `init --ide <other>` 默认失败或提示改用迁移。
- `sync` 拒绝不存在的项目根。
- `sync/verify/doctor` 对非法 `--ide` 输出友好错误，并不写任何文件。
- 所有命令对 `--project <file>` 输出明确错误，且不生成或修改其他目录。
- `verify` 对 `project.config.yaml` 做 schema 校验，包括 pack、ide、平台枚举值和 workflow 类型。
- `verify` 能发现嵌套 ai-video-workflow 项目，至少给出 warning 级 issue 或 doctor 提示。
- CLI action 抛错时不打印 Node stack；需要 stack 时用显式 debug 模式。
- `verify/doctor` 对不存在的项目根输出单一明确诊断。
- `export-obsidian` 拒绝不存在项目、缺配置项目、缺 Step 目录项目和零源文件项目。
- `export-obsidian` 对 `verify` 已失败的项目默认拒绝，或明确要求 `--allow-partial` 并在 manifest/dashboard 标注 partial。
- `verify-obsidian` 对空源项目、错误项目根和 manifest 源文件缺失失败。
- `export-obsidian/verify-obsidian` 对 `--out/--vault <file>` 输出明确错误。
- `export-obsidian --out <project>/<普通子目录>` 默认失败；项目内观看层只能走 `--in-project-view` 解析到 `_views/obsidian`。
- `export-obsidian --force --dry-run` 对含 `.git` 的目标也报告安全阻止原因。
- 外部 vault `--force` 对非空且无 manifest 的目录要求确认或失败。
- `sync --project <工具仓库根或源码目录>` 默认失败或明确提示这不是创作项目。
- README、中文快速开始、英文快速开始、VitePress sidebar 中的初始化入口保持一致。
- 文档版本叙事有测试或审查项，避免 package version、路线图版本和功能批次混在一起。

按 8 类排查面映射后，后续测试应至少覆盖：

- 交互式 `init`：真实 TTY/模拟 prompt 的默认值、空值、取消、非 TTY 失败和成功后 next steps 输出。
- 路径与文件系统边界：安全名称校验、路径逃逸、普通文件目标、权限/只读/联接目录和错误输出格式。
- 已有目录分类：空目录、非空素材目录、已有 Git 仓库、已有宿主文件、已有旧 runtime 和已有 Step 目录。
- 重复操作与幂等性：重复 `init` 默认拒绝，重复 `sync` 幂等，多 IDE sync 不改变主配置，中断半成品可诊断。
- Obsidian 输出目录风险：不存在项目拒绝导出，partial 项目默认拒绝或显式标注，普通文件 vault 失败，dry-run 也报告 `.git` 阻止。
- 配置文件可信度：缺字段、非法枚举、错误类型、未知 pack、YAML 语法错误和旧 schema 版本都变成 verify/doctor issue。
- 新手第一步体验：生成项目 README 和 Step 1 起步文件非空，初始化完成输出“打开哪个目录”和“第一句话”，中英文入口同步。
- 测试覆盖映射本身：每个 P0/P1 风险在测试中至少有一个“失败且不写文件”或“诊断清晰”的回归断言。

## 后续封堵规划记录

本节是问题封堵方向记录，不是实施计划。真正进入代码实现前，仍需按仓库 `AGENTS.md` 单独写详细实施计划文档。

后续应按“先挡高风险误写，再改善新手体验”的顺序推进。

第一层：统一路径和名称预检。

- 所有会写文件的命令先确认输入路径类型：目标项目根必须是目录，不能是普通文件，不能是不存在路径，除非命令语义明确是创建新项目。
- `init --name` 和 `new-pack --name` 复用同一个安全名称校验：禁止空值、全空白、首尾空白、`.`、`..`、路径分隔符、盘符、UNC、绝对路径、Windows 保留名、尾随点、尾随空格、控制字符和不可见字符。
- 保留中文、正常空格、短横线、下划线和常见可读名称，但所有输出命令示例必须自动加引号。
- 初始化前向上查找 `project.config.yaml`，如果当前目录已经在某个 ai-video-workflow 项目内部，默认拒绝再创建嵌套项目。
- 目标目录已存在且非空时，先分类：已有 ai-video-workflow 项目、已有 Git 仓库、普通素材目录、智能体宿主目录、未知目录。默认不覆盖，给出摘要和下一步建议。

第二层：统一 CLI 错误体验。

- 所有 commander action 使用统一错误捕获，默认输出人类可读的 `Error: ...`，不打印 Node stack。
- 对非法 `--ide`、`--image`、`--video` 给出可选值，必要时给出拼写建议。
- 对缺少 `--out/--vault`、目标是文件、路径不存在、配置 YAML 语法错误等，输出具体原因和下一步，而不是底层 ENOTDIR/EEXIST/YAML stack。
- 需要 stack 时，后续可以引入显式 debug 模式，避免新手默认看到内部实现路径。

第三层：项目有效性和重复操作语义。

- 引入 `project.config.yaml` schema 校验，检查 pack、ide、平台枚举值和 workflow 类型。
- `verify/doctor` 增加“项目根不存在”“项目根不是目录”“嵌套项目”“配置 schema 错误”“共享文档版本漂移”等明确 issue。
- `sync` 只对有效项目补 runtime，不再对不存在路径创建半成品目录。
- `sync` 应识别工具仓库根、`apps/cli`、pack 目录和其他源码目录，避免把创作项目 runtime 写入工具代码区。
- 重复 `init` 默认失败；如果目标已经是项目，提示使用 `verify`、`doctor`、`sync` 或后续专门的迁移命令。
- 多 IDE 模式需要明确表达：`sync --ide <second-ide>` 是增加 runtime，不等于修改主 IDE；修改主 IDE 应有显式迁移语义，而不是重复 `init`。

第四层：生成项目的新手入口。

- `init` 成功后输出项目绝对路径、下一步打开目录、推荐对智能体说的话、verify 命令和 Obsidian 观看层说明。
- 生成项目 README 改成创作者首页，明确“这是创作项目，不是工具仓库”。
- Step 1 到 Step 5 不应全部为空；至少提供可填写的 Step 1 起步文件，或在 README 中明确指向 runtime templates 的复制方式。
- 如果目标目录已有素材，向导应先盘点并建议把素材放入合适 Step，而不是直接初始化覆盖同名文件。

第五层：Obsidian 和 MCP 边界封堵。

- `export-obsidian` 默认要求项目结构有效；如果 `verify` 失败，应拒绝导出或要求显式 partial 模式。
- 不存在项目、普通文件项目、零源文件项目、缺 Step 目录项目不能生成“看似成功”的 vault。
- 除 `--in-project-view` 解析出的 `_views/obsidian` 外，`--out` 不应指向项目内部普通子目录；尤其不能在项目内部普通子目录上执行 `--force` 清理。
- `verify-obsidian` 需要确认源项目存在且 manifest source entries 对应真实 Step 文件；空源项目不能默认通过。
- `--force --dry-run` 也应执行 `.git` 安全预检，并告诉用户真实执行会被阻止。
- MCP context/server 保持只读，但错误信息应指向 `verify/doctor/init/sync` 的正确修复路径。

第六层：文档入口和版本叙事。

- README、README.zh-CN、quickstart、VitePress sidebar 应统一指向“新手让智能体代跑 CLI”的入口。
- 英文文档需要有与中文一致的用户入口；当前中文记录文档可保留为审计材料，后续可拆出正式双语新手指南。
- 版本叙事应降速，区分 package version、功能批次、历史计划编号和对外发布版本，避免继续从 v0.7 往上推造成 1.0 临近错觉。

## 推荐的顺序策略

最稳妥的默认顺序是：

1. 智能体先在工具仓库中检查 CLI 是否可用。
2. 智能体询问用户创作项目要放在哪个父目录。
3. 智能体确认目标子目录不存在或为空。
4. 智能体运行 CLI 初始化。
5. 用户或智能体切换到生成后的创作项目目录。
6. 智能体读取创作项目里的 `AGENTS.md` 和 `docs/ai-workspace/`。
7. 智能体带用户从 Step 1 开始，而不是继续在工具仓库里写创作内容。

对于已经有内容的目录，默认顺序应改成：

1. 先只读盘点目录。
2. 判断这是工具仓库、创作项目、普通素材目录，还是某个智能体已经使用过的工作目录。
3. 如果已经是 ai-video-workflow 项目，优先运行 `verify/doctor`。
4. 如果只是缺少 runtime 规则，优先运行 `sync`。
5. 只有确认用户要把该目录改造成 ai-video-workflow 创作项目时，才运行初始化。

## 是否完全交给智能体

不建议完全取消 CLI，也不建议让智能体手工复制模板来初始化项目。

更稳妥的分工是：

- 智能体负责询问用户、解释选项、降低心理门槛。
- CLI 负责确定性地创建项目、同步规则、写入 IDE runtime、生成配置和执行校验。

这样既能照顾没有编程经验的用户，也能保留自动化、测试、CI、文档复现和高级用户脚本化使用。

## 是否保留用户手动初始化

需要保留。

原因：

- 熟练用户需要非交互命令，便于批量创建项目或写文档示例。
- 测试和 CI 需要可复现的脚本化入口。
- 智能体最终也应该调用同一套 CLI，而不是绕开 CLI 自己造项目。
- CLI 是项目结构和 adapter 同步的权威实现，不能只存在于智能体提示词里。

但文档默认入口应从“用户手敲完整 CLI 参数”调整为“用户让智能体代跑初始化”。手动初始化应作为高级用法保留，而不是面向零编程用户的第一入口。

## 当前产品判断

短期不需要做桌面软件、Obsidian 插件或更重的平台层。更直接的改进是把初始化体验包装好：

- 仓库根入口明确说明工具仓库和创作项目的区别。
- 快速开始给出新手可复制的第一句话。
- `init` 完成后输出下一步路径和智能体提示词。
- 生成项目 README 改成真正的项目首页。
- 智能体规则补充“遇到新手时先问问题，不要要求用户自己拼命令”。
- CLI 参数增加更友好的错误提示和确认步骤。

这类改进可以作为单独的小版本或体验迭代处理，不需要提高到 1.0 叙事。
