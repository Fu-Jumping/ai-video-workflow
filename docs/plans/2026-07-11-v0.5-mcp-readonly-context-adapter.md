# v0.5 MCP Read-Only Context Adapter Implementation Plan

> **For Codex:** REQUIRED SUB-SKILL: Use executing-plans to implement this plan task-by-task.

**Goal:** Add an MCP adapter that exposes `ai-video-workflow` projects as read-only resources, prompts, and diagnostics without making MCP a second source of truth or a default writer.

**Architecture:** Start with a contract and docs slice, then build a reusable read-only context model, then add a local stdio MCP server only after the resource and prompt surfaces are covered by tests. MCP must read project Step files and pack metadata, expose copy-ready handoff prompts, and offer only read/verify tools in v0.5. Any write, reverse sync, generation-platform call, or LibTV action requires a separate later plan.

**Tech Stack:** TypeScript CLI, Vitest, existing `agent-adapter-contract` schema, optional `@modelcontextprotocol/sdk` only in the server slice, Markdown docs, VitePress.

---

## Baseline

The repository already has:

- v0.3 Obsidian vault projection, incremental export, manifest protection, dashboard, review maps, shot pages, agent handoff, optional `.obsidian` UI suggestions, and real-vault QA.
- v0.4 agent adapter contract and hardened Codex, Cursor, Claude Code, and Trae runtime mirrors.
- `verify --ide <id>` runtime checks for Codex, Cursor, Claude Code, and Trae.
- `doctor` suggestions that point missing runtime files to the matching `sync --ide <id>` command.
- `schemas/agent-adapter-contract.schema.json` with `read-only-context` as an allowed sync direction.
- `apps/cli/tests/fixtures/agent-adapters/` fixtures for Codex, Claude Code, and Trae.

The repository does not yet have:

- `mcp.contract.json`.
- MCP contributor docs beyond short references in adapter boundary docs.
- Shared code for producing a stable project context index.
- An MCP CLI command or server entrypoint.
- `@modelcontextprotocol/sdk` dependency.

## Product Decision

MCP v0.5 starts as a read-only context adapter.

It should help an agent answer:

- What project is this?
- Which pack and Step files define the workflow?
- Which shots exist?
- Which Step 3, Step 4, Step 5, and Step 6 files belong to each shot?
- Which file should be edited for story, image prompt, motion, or execution-plan changes?
- Which verification commands should run after edits?

It should not:

- edit Step files
- export or mutate Obsidian vaults
- sync IDE runtime mirrors
- call generation platforms
- call LibTV
- write `.obsidian/`
- write `.codex/`, `.cursor/`, `.claude/`, or `.trae/`
- maintain execution state as upstream truth

## External MCP Assumptions

The official MCP model exposes three server-side capability types:

- resources: readable context surfaces
- prompts: reusable prompt templates
- tools: callable functions

For this repository:

- resources should be the primary surface
- prompts should be handoff templates that point to source Step files
- tools should initially be read-only diagnostics, such as project summary and verify result
- stdio transport is enough for a first local server
- remote transport, auth, sampling, elicitation, and write tools are out of scope for v0.5

## Non-Goals

- No MCP write tool in v0.5.
- No automatic Step file patching from MCP.
- No reverse sync from Obsidian, IDE runtime mirrors, or MCP resources.
- No LibTV work.
- No generation-platform execution.
- No remote hosted MCP service.
- No MCP client implementation.
- No dynamic plugin system for user-defined MCP resources.
- No migration of `Ide` to include `mcp`; MCP is an adapter contract and server surface, not an IDE runtime mirror.

### MCP Prompts

Expose at least these prompts:

```text

revise_image_prompt
revise_video_prompt
verify_project
```

Prompt arguments:

- `shotId` for shot-level prompts where applicable
- optional `focus` string for user intent

Prompt rules:

- instruct the agent to read source Step files
- instruct the agent to edit only source Step files when changes are needed
- warn that Obsidian, IDE runtime mirrors, and MCP resources are generated or read-only surfaces
- include verification commands

### MCP Tools

Expose only read-only tools in v0.5:

```text
get_project_summary
list_shots
get_shot_context
run_project_verify
```

Tool behavior:

- `get_project_summary`: returns the same core object as `mcp-context`
- `list_shots`: returns shot ids, titles, source paths, status fields if available
- `get_shot_context`: returns Step 3/4/5/6 source path references and short excerpts for one shot
- `run_project_verify`: runs existing verification logic and returns issues; it must not repair anything

## Architecture Decisions

### Decision 1: Contract Before Server

Write the MCP contract fixture and docs first. This keeps the adapter boundary explicit before adding a server dependency.

### Decision 2: Shared Context Builder

Create a small internal context builder before server code:

```text
apps/cli/src/lib/mcp/context.ts
```

The server and CLI command both call this builder. Tests should cover the context builder without starting a protocol server.

### Decision 3: Read-Only Tools Only

Tools can call read-only local logic, including `verifyProject`. They cannot call `syncProject`, `exportObsidianVault`, `fs.writeFile`, or generation-platform APIs.

### Decision 4: No `Ide = "mcp"`

`Ide` currently means a project runtime mirror target. MCP is not a runtime mirror and should not be added to `SUPPORTED_IDES` in v0.5. Use a separate adapter fixture and MCP commands.

### Decision 5: Deterministic Output

All context output should be sorted by stable keys:

- Step number ascending
- shot order if available
- path lexicographic fallback
- prompt/tool/resource names lexicographic where no domain order exists

This makes tests reliable and makes MCP responses easy to diff.

## Security and Safety Checks

- MCP server must not write project files.
- MCP tools must not invoke `syncProject`, `exportObsidianVault`, or write APIs.
- MCP responses must use project-relative paths.
- MCP context must not include local absolute project root paths.
- MCP server must be scoped to the explicit `--project` path.
- Missing or invalid files should be reported as diagnostics, not repaired.
- Prompt templates must say source Step files are the editable truth.
