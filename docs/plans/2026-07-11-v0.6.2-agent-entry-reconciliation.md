# v0.6.2 Agent Entry Reconciliation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让 Cherry Studio 先创建工作目录、其他智能体先创建工作目录、以及两者先后交替进入同一目录时，都能稳定合并 `AGENTS.md`、`docs/ai-workspace/`、`SOUL.md`、`USER.md` 和 `memory/` 的入口边界。

**Architecture:** 当前 v0.6/v0.6.1 已经做到“不生成、不覆盖、不校验污染”Cherry Studio 宿主文件；v0.6.2 继续补上“入口冲突识别与合并指导”。实现上把共享入口 marker、Cherry 宿主表面、合并片段、verify/doctor 诊断集中到可复用 helper，避免 `sync.ts`、`verify.ts`、docs 和测试各写一套分散规则。

**Tech Stack:** TypeScript CLI, fs-extra, Vitest, VitePress docs, Markdown plan/docs, existing `syncProject`, `verifyProject`, and `diagnoseProject` flows.

---

## 背景与问题

当前两种顺序的实际行为：

1. Cherry Studio 先进入项目根：
   - 可能自动创建 `SOUL.md`、`USER.md`、`memory/`。
   - 之后 `sync` 会创建 `AGENTS.md` 和 `docs/ai-workspace/`。
   - 只要没有自定义 `AGENTS.md`，不会冲突。

2. Codex/Cursor/Claude Code/Trae 先进入项目根：
   - `sync` 会创建 `AGENTS.md` 和 `docs/ai-workspace/`。
   - 之后 Cherry Studio 创建 `SOUL.md`、`USER.md`、`memory/`。
   - 当前 verify 会跳过这些宿主文件，基本不会冲突。

真正需要优化的是第三类边界：

- Cherry Studio 或用户已经创建了自定义 `AGENTS.md`。
- 其他智能体之后执行 `sync`，因为保护用户文件，不会覆盖它。
- `verify` 会报 `invalid-shared-agent-entry`，但目前不能明确告诉用户“保留你的 Cherry/用户入口，只把 ai-video-workflow 合并块加入 AGENTS.md”。

## Security And Consistency Invariants

从安全加固角度，本阶段必须保持这些不变量：

- `sync` 不读取、不解析、不重写 `SOUL.md`、`USER.md` 或 `memory/` 内容。
- `verify` 可以检测这些宿主表面是否存在，但不得扫描其正文、不得把其中的本机路径或私人记忆输出到诊断。
- `AGENTS.md` 是唯一允许被读取并诊断 marker 的根入口文件，因为它是跨智能体共享入口。
- `docs/ai-workspace/` 是共享 AI 文档层，允许由 CLI write-if-missing 初始化。
- `project-step-files` 仍是创作事实源。
- Cherry Studio host memory/persona surfaces are not project truth by default.
- 任何自动修复都不得覆盖已有用户 `AGENTS.md`；只能生成合并建议或单独的 merge guide。
