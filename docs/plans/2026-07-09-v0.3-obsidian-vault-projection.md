# v0.3 Obsidian Vault Projection Implementation Plan

> **For Codex:** REQUIRED SUB-SKILL: Use executing-plans to implement this plan task-by-task.

**Goal:** Build an Obsidian vault projection for `ai-video-workflow` projects so creators can read, navigate, review, and visually manage Step 1 to Step 6 without treating Obsidian as a second workflow source.

**Architecture:** Keep `packs/official-ai-video/` and the project Markdown files as the source of truth. Add a one-way Obsidian adapter that exports a generated vault view with enriched properties, tags, dashboards, Bases, Canvas maps, and relative internal links. Verification must prove the projection is portable, traceable to source files, and does not weaken Step 3 / Step 4 alignment or Step 4 file contracts.

**Tech Stack:** TypeScript, Commander, fs-extra, yaml, Vitest, VitePress, Markdown, YAML frontmatter, Obsidian core plugins, JSON Canvas, Obsidian Bases.

---

## Scope

This plan implements the first Obsidian milestone after v0.2.

It focuses on:

- exporting a generated Obsidian-friendly vault from an existing project
- using Obsidian core features for viewing and navigation
- adding properties and nested tags that make notes queryable
- generating dashboards instead of relying on the file tree
- generating Bases for table/card/list views
- generating JSON Canvas maps for workflow and shot relationships
- validating the generated vault projection
- documenting how creators should use the Obsidian view

It deliberately does not implement:

- an Obsidian community plugin
- direct editing sync from Obsidian back to source project files
- LibTV canvas execution
- MCP server runtime
- direct image or video generation
- automatic installation or enabling of community plugins
- writing `.obsidian/` UI state by default

## Research Findings

The design is based on the Obsidian official help vault, JSON Canvas spec, and representative community plugin documentation.

### Official Obsidian capabilities to use in v0.3

- **Internal links:** Obsidian supports both wikilinks and Markdown links. Markdown links are more portable, but wikilinks give compact authoring. The projection should generate Markdown links in source-facing docs and may generate wikilinks inside Obsidian-only dashboard files where they improve display.
- **Backlinks and outgoing links:** These only become useful when projection files contain real internal links. The adapter must link concept, setting, storyboard, image prompt, video prompt, execution plan, and generated index pages instead of leaving them as isolated copied files.
- **Properties:** Obsidian stores structured note metadata in YAML frontmatter. Supported property types include text, lists, numbers, checkboxes, dates, tags, and links. Internal links in properties must be quoted.
- **Tags:** Tags can be stored in the `tags` property as a list. Nested tags use `/`, which is useful for `ai-video/step/03-storyboard`, `ai-video/status/ready`, and `ai-video/shot/shot-001`.
- **Search:** Search supports `tag:`, `path:`, `file:`, `content:`, task operators, and property search. Search results can be embedded with `query` code blocks.
- **Graph view:** Graph view visualizes notes and internal links. The adapter should create enough real links for local graph and global graph to show workflow relationships.
- **Canvas:** Canvas is a core plugin. `.canvas` files use the open JSON Canvas format with `nodes` and `edges`; nodes can be text, file, link, or group; edges can be labeled and colored. The adapter can generate `.canvas` files directly without an Obsidian plugin.
- **Bases:** Bases is a core plugin for database-like views of notes. A `.base` file is YAML with filters, formulas, properties, summaries, and views. Views can render tables, cards, lists, and maps; cards can show cover images from a property.
- **Templates:** Templates are useful for human follow-up notes, but v0.3 should export starter templates as optional Obsidian notes, not depend on a plugin configuration.
- **Bookmarks and Workspaces:** These are useful UI features but represent local application state. v0.3 should document manual setup rather than write `.obsidian/` files by default.

### Community plugin findings to keep optional

- **Dataview:** Can query frontmatter and inline fields with DQL and JavaScript. It remains useful for advanced users, but Bases now covers the first-party database need for v0.3.
- **Tasks:** Can query and toggle tasks across a vault. v0.3 can include optional task-query recipes, but the core projection should work without it.
- **Kanban:** Creates Markdown-backed Kanban boards. Useful for later project management views, but not required for the first Obsidian adapter.
- **Excalidraw:** Offers richer visual thinking than core Canvas, with internal links and Markdown embeds, but it is a large community plugin. v0.3 should not depend on it.
- **Community plugin security:** Community plugins run third-party code with broad local capabilities. The default Obsidian adapter must be core-plugin-only.

### Local Windows validation notes

- Prefer writing `.canvas` files directly instead of sending complex JSON through the Obsidian CLI.
- Avoid concurrent Obsidian CLI calls during manual validation.
- Treat Obsidian CLI or UI checks as optional manual QA; automated tests should validate generated files without requiring Obsidian to launch.

## Design Decisions

1. **One-way projection.** The generated vault is a view over project files. Users can inspect and annotate it, but v0.3 does not sync edits back into source Step files.

2. **Generated files must declare provenance.** Every generated note copied or transformed from a source file must include frontmatter with `source_path`, `source_kind`, and `projection_generated: true`.

3. **Core-plugin-first.** Dashboard Markdown, internal links, tags, Search query blocks, Bases, and Canvas are enough for the first release. Dataview, Tasks, Kanban, and Excalidraw are documented as optional recipes only.

4. **No hidden vault mutation.** The CLI must require `--out <path>` for the export destination. It must refuse to overwrite existing generated output unless `--force` is passed.

5. **Readable first, queryable second.** The user experience starts at a generated dashboard and canvas, then supports Bases and tags for structured browsing. The file tree is only a fallback.

6. **Relative links only.** Generated Markdown links and JSON Canvas file references must be relative to the generated vault root. No drive paths, `file://`, `vscode://`, or app-specific URIs.

7. **Adapter boundary remains strict.** Obsidian may add properties, index files, and visual projections, but it cannot redefine Step 1 to Step 6 or modify Step 4 contracts.

## Generated Properties Contract

Every generated workflow note should include frontmatter similar to:

```yaml
---
projection_generated: true
workflow_pack: official-ai-video
project: official-mini-film
source_path: 03_storyboard/shot-001.md
source_kind: storyboard
step: 3
step_name: storyboard
shot_id: shot-001
status: ready
tags:
  - ai-video/project
  - ai-video/step/03-storyboard
  - ai-video/type/storyboard
  - ai-video/shot/shot-001
  - ai-video/status/ready
---
```

Use quoted internal links for link-valued properties:

```yaml
image_prompt: "[[Shot 001 - Image Prompt]]"
video_prompt: "[[Shot 001 - Video Prompt]]"
```

## Obsidian Viewing Experience

The exported vault should be usable from these entry points:

1. **Project Home:** human-readable overview, workflow progress, links to dashboards, embedded Bases, and embedded Canvas.
2. **Review Dashboard:** query blocks and links for missing status, review-needed notes, and currently ready execution items.
3. **Shot Index:** one page per shot that links storyboard, image prompt, video prompt, execution plan, and any generated media placeholders.
4. **Production Board:** status-first view for what can be generated, what is blocked, and what needs review.
5. **Bases:** table/card/list views over properties rather than hand-written tables only.
6. **Canvas:** visual pipeline from Step 1 to Step 6 and shot-level frame alignment map.
7. **Graph and Backlinks:** generated links should make Obsidian's native graph and backlinks meaningful without extra plugins.

**Step 3: Implement scanner**

Create `apps/cli/src/lib/obsidian/scan.ts`:

```ts
import fs from "fs-extra";
import path from "node:path";

import type { ObsidianSourceFile, ObsidianSourceKind } from "./types.js";

const stepDirs: Array<{ dir: string; step: number; sourceKind: ObsidianSourceKind }> = [
  { dir: "01_concept", step: 1, sourceKind: "concept" },
  { dir: "02_setting", step: 2, sourceKind: "setting" },
  { dir: "03_storyboard", step: 3, sourceKind: "storyboard" },
  { dir: "04_image_prompts", step: 4, sourceKind: "image-prompt" },
  { dir: "05_video_prompts", step: 5, sourceKind: "video-prompt" },
  { dir: "06_execution_plan", step: 6, sourceKind: "execution-plan" }
];

function inferShotId(fileName: string): string | undefined {
  const match = fileName.match(/shot[-_ ]?(\d+)/i);
  return match ? `shot-${match[1].padStart(3, "0")}` : undefined;
}

function titleFromFileName(fileName: string): string {
  return path.basename(fileName, ".md").replace(/[-_]+/g, " ").replace(/\b\w/g, (char) => char.toUpperCase());
}

export async function scanProjectForObsidian(projectRoot: string): Promise<ObsidianSourceFile[]> {
  const files: ObsidianSourceFile[] = [];
  for (const stepDir of stepDirs) {
    const fullDir = path.join(projectRoot, stepDir.dir);
    if (!(await fs.pathExists(fullDir))) {
      continue;
    }
    const entries = (await fs.readdir(fullDir)).filter((name) => name.endsWith(".md")).sort();
    for (const entry of entries) {
      files.push({
        sourcePath: path.join(stepDir.dir, entry),
        sourceKind: stepDir.sourceKind,
        step: stepDir.step,
        title: titleFromFileName(entry),
        shotId: inferShotId(entry)
      });
    }
  }
  return files;
}
```
