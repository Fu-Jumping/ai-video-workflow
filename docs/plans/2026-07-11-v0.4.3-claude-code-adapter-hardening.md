# v0.4.3 Claude Code Adapter Hardening Implementation Plan

> **For Codex:** REQUIRED SUB-SKILL: Use executing-plans to implement this plan task-by-task.

**Goal:** Harden the Claude Code adapter against the agent adapter contract so Claude Code receives the same workflow mirror, skill bundles, and boundary rules as Codex and Cursor.

**Architecture:** Keep `packs/official-ai-video/` and project Step 1 to Step 6 files as the only workflow truth. Treat `CLAUDE.md` and `.claude/` as generated Claude Code runtime entrypoints, with `.claude/ai-video-workflow/` as a read-only workflow mirror and `.claude/skills/` as native skill bundles.

**Tech Stack:** TypeScript CLI, Vitest, Markdown docs, YAML sync rules, Claude Code `CLAUDE.md`, `.claude/skills/`, `.claude/commands/`, and `.claude/ai-video-workflow/` runtime files.

## Scope

This milestone turns the existing Claude Code adapter from a minimal `.claude/skills` copy into a full adapter-contract runtime mirror.

In scope:

- Generate `CLAUDE.md` as the Claude Code root entrypoint.
- Generate `.claude/skills/` from pack skill bundles.
- Generate `.claude/ai-video-workflow/` with overview, longform skills, skill bundles, templates, workflow docs, and indexes.
- Generate `.claude/commands/ai-video-workflow.md` as a concise command-style handoff entry.
- Update sync rules and Claude Code docs to match actual CLI output.
- Add tests that prove required files exist and generated runtime text does not contain absolute local links.

Out of scope:

- No Claude-specific workflow reinterpretation.
- No automatic edits to Step files.
- No reverse sync from `.claude/` to project source.
- No `.obsidian/` writes.
- No MCP server or LibTV execution work.
- No dependency on undocumented Claude Code internals beyond stable file entrypoints.

## Current Baseline

Current implementation in `apps/cli/src/lib/sync.ts`:

```ts
async function syncClaudeCode(projectRoot: string, packRoot: string): Promise<void> {
  await copyDirectory(path.join(packRoot, "skills"), path.join(projectRoot, ".claude", "skills"));
  await writeFileIfMissing(path.join(projectRoot, "CLAUDE.md"), "# Claude Code Compatibility Entry\n");
}
```

Current docs already expect a richer adapter shape:

- `packs/official-ai-video/ide-integrations/claude-code/README.md` recommends `.claude/ai-video-workflow/`, `.claude/skills/`, `.claude/commands/`, and `CLAUDE.md`.
- `docs/en/ide-integrations/claude-code.md` and `docs/zh/ide-integrations/claude-code.md` describe the adapter contract but do not list exact generated mirror files.
- `packs/official-ai-video/checks/sync-rules.yaml` currently records only `skillsDir: ".claude/skills"` for `claude-code`.

## Design Decisions

- Use `CLAUDE.md` as the stable root entrypoint because Claude Code commonly reads it at project root.
- Use `.claude/skills/` for directory-style skill bundles copied from `packs/official-ai-video/skills/`.
- Use `.claude/ai-video-workflow/` as the full runtime mirror, matching the adapter-contract pattern already used by Codex and Cursor.
- Include `packs/official-ai-video/workflow/` in the Claude mirror because the pack README explicitly lists workflow content as part of the desired Claude runtime mirror.
- Add a single `.claude/commands/ai-video-workflow.md` entry instead of many step-specific command files in this slice.
- Use `writeFileIfMissing` for `CLAUDE.md` and command entrypoints so user-written local Claude guidance is preserved.
- Use `copyDirectory` for generated mirror folders because these are adapter-owned runtime copies.
- Keep all generated content generic and relative-link-only.
